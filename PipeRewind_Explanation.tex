\section{The Problem: Pipelines Are a Black Box}
\label{sec:problem}
% ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

\subsection{What is a shell pipeline?}

In Linux, you can chain programs together with the pipe character
(\texttt{|}). The output of one program automatically becomes the input
of the next:

\begin{lstlisting}[language=bash]
cat access.log | grep ERROR | sort | uniq -c
\end{lstlisting}

This single line runs \emph{four} programs at the same time:

\begin{enumerate}[leftmargin=18pt]
    \item \textbf{cat} reads a log file and prints every line.
    \item \textbf{grep} keeps only lines containing ``ERROR''.
    \item \textbf{sort} alphabetizes those lines.
    \item \textbf{uniq -c} counts how many times each line appears.
\end{enumerate}

Data flows left to right through invisible connections called
\emph{pipes}. Each pipe is a one-way tunnel managed by the operating
system.

\subsection{Why is debugging hard?}

Imagine the command above prints nothing. Where is the bug?

\begin{itemize}[leftmargin=18pt]
    \item Maybe \texttt{cat} could not find the file.
    \item Maybe \texttt{grep} filtered out everything.
    \item Maybe the data reached \texttt{sort} but not \texttt{uniq}.
\end{itemize}

You cannot see what flows between programs because the data travels
through the kernel, invisible to the user. This is the core problem
PipeRewind solves.

    \includegraphics[width=0.75\textwidth]{images/pipeline_problem.png}
    \caption{The data moving between commands is invisible --- you only see the final output.}
    \label{fig:problem}
\end{figure}

\section{The Solution: PipeRewind}
\label{sec:solution}
% ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PipeRewind is a tool written in C that sits between every pair of
commands in a pipeline. Think of it like a security camera for your
data: it records every piece of text that passes through, stamps it
with the exact time, and saves everything to a file. Later, you can
``rewind'' and replay the entire execution, stepping through it one
event at a time --- just like scrubbing through a video.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.75\textwidth]{images/piperewind_solution.png}
    \caption{PipeRewind makes everything visible by recording the data between each command.}
    \label{fig:solution}
\end{figure}

\begin{notebox}[Key Idea]
PipeRewind is \textbf{transparent}. It does not change how your
pipeline works. The exact same output appears on your screen --- but
now a complete recording is saved on disk for later inspection.
\end{notebox}

\section{How to Use It (Step by Step)}
\label{sec:usage}
% ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

\subsection{Step 0: Build}

You need a Linux machine with GCC and the ncurses library installed.
Then just type:

\begin{lstlisting}[language=bash]
make clean && make
\end{lstlisting}

This compiles all seven source files and produces a single executable
called \texttt{piperewind}.

\subsection{Step 1: Record a pipeline}

\begin{lstlisting}[language=bash]
./piperewind record "echo -e 'banana\napple\ncherry' | sort | head -2"
\end{lstlisting}

You will see the normal output (\texttt{apple} and \texttt{banana}) on
your screen, and a file called \texttt{trace.prt} is created silently
in the background.

\begin{tipbox}[Tip]
Use \texttt{-v} to see progress messages, or \texttt{-o my\_trace.prt}
to choose a different file name.
\end{tipbox}

\subsection{Step 2: Inspect the recording}

The simplest way to look at a trace is the \texttt{dump} command, which
prints a plain-text timeline:

\begin{lstlisting}[language=bash]
./piperewind dump trace.prt
\end{lstlisting}

Example output (abbreviated):

\begin{lstlisting}
[  0.07 ms] stage 0  PROC_START
[  1.38 ms] stage 0  DATA_OUT  20 bytes  "banana\napple\ncherry\n"
[  1.61 ms] stage 1  DATA_IN   20 bytes  "banana\napple\ncherry\n"
[  2.44 ms] stage 1  DATA_OUT  20 bytes  "apple\nbanana\ncherry\n"
[  2.58 ms] stage 2  DATA_OUT  13 bytes  "apple\nbanana\n"
[  2.80 ms] stage 2  PROC_EXIT exit_code=0
\end{lstlisting}

Reading this from top to bottom tells you the complete story: Stage~0
produced the unsorted fruit list, Stage~1 (sort) received it and output
the sorted version, and Stage~2 (head) kept only the first two lines.

\subsection{Step 3: Open the visual replay viewer}

For a richer experience, open the interactive TUI:

\begin{lstlisting}[language=bash]
./piperewind replay trace.prt
\end{lstlisting}

\subsection{Step 4 (optional): Watch a pipeline live}

Instead of recording first and replaying later, you can do both at the
same time:

\begin{lstlisting}[language=bash]
./piperewind live "tail -f /var/log/syslog | grep error"
\end{lstlisting}

The viewer opens immediately, and events appear as the pipeline
produces them. Press \texttt{f} to toggle auto-scrolling.

Let's walk through each file:

\subsection{main.c --- The front door}

This is where the program starts. It reads what you typed on the
command line (\texttt{record}, \texttt{replay}, \texttt{live}, or
\texttt{dump}) and calls the right function. Think of it as a
receptionist that directs you to the right office.

\subsection{pipeline.c --- Understanding your command}

When you write \texttt{"echo hello | sort"}, this file splits that
string into separate commands. It is smart enough to handle tricky
cases like single quotes (\texttt{'hello world'}), double quotes, and
backslash escapes. Under the hood it uses a \emph{state machine} ---
a pattern where the code keeps track of whether it is currently inside
quotes or not, and adjusts its behavior accordingly.

\subsection{capture.c --- The recording engine}

This is the heart of PipeRewind. When you run \texttt{record}, this
file:

\begin{enumerate}[leftmargin=18pt]
    \item Creates child processes (one per command) using \texttt{fork}
          and \texttt{exec}.
    \item Connects them with pipes, but inserts itself in the middle
          of each pipe so it can see the data.
    \item Uses Linux's \texttt{epoll} system to efficiently wait for
          data from \emph{any} of the pipes without busy-waiting.
    \item Every time data arrives, it forwards the bytes to the next
          command and simultaneously writes a timestamped record to
          the trace file.
\end{enumerate}

The result is completely transparent: the pipeline behaves exactly the
same as if PipeRewind were not there.

\subsection{trace.c --- The filing cabinet}

This file handles reading and writing the binary trace file
(\texttt{.prt}). It stores a compact header describing the pipeline,
followed by a stream of events, followed by an index table for fast
lookups. Like a book's table of contents, the index lets the viewer
jump to any point in time without reading everything before it.

It also supports \emph{live refresh}: while the capture engine is
still writing new events, the reader can detect and load them
dynamically.

\subsection{tui.c --- The visual viewer}

The largest file in the project. It uses the \texttt{ncurses} library
to draw a full-screen terminal interface with:

\begin{itemize}[leftmargin=18pt]
    \item A header bar showing the file name and event count.
    \item A timeline showing how far through the recording you are.
    \item A list of pipeline stages with color-coded status (green for
          running, red for exited, yellow for blocked).
    \item A data panel showing the actual bytes that flowed at the
          selected moment in time.
\end{itemize}

It loads events through a \emph{sliding window}: instead of putting
the entire trace into memory, it keeps only a chunk of about 20{,}000
events around your current position, loading more as you scroll.

\subsection{diff.c --- Spot the difference}

When you press \texttt{d} in the viewer, this file compares the output
of one stage with the input of the next stage, line by line. It uses
a well-known algorithm called \emph{Longest Common Subsequence} (LCS)
to find which lines were added, removed, or changed --- the same idea
used by \texttt{git diff}.

\subsection{procstate.c --- Health monitor}

This tiny file reads each process's status from the Linux
\texttt{/proc} filesystem (the kernel's real-time status board). It
reports whether a process is Running, Sleeping, or Blocked, which
helps you find bottlenecks.

\section{How Data Flows Through PipeRewind}
\label{sec:dataflow}

pipeline like \texttt{echo hello | cat}:

When data arrives from \texttt{echo}:

\begin{enumerate}[leftmargin=18pt]
    \item The engine writes a \textbf{DATA\_OUT} event (``Stage~0
          produced these bytes'').
    \item It writes a \textbf{DATA\_IN} event (``Stage~1 received
          these bytes'').
    \item It forwards the bytes to \texttt{cat}'s standard input.
\end{enumerate}

This three-step process repeats for every chunk of data, on every
pipe, creating a complete timeline of the pipeline's life.

\section{The Trace File Format}
\label{sec:trace}
% ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The \texttt{.prt} file is a custom binary format designed for fast
random access. Here is how it is laid out:

\begin{figure}[H]
\centering
\begin{tikzpicture}[
    block/.style={draw=prgray!50, fill=#1, minimum width=8cm,
                  minimum height=1cm, font=\small, text=prgray,
                  rounded corners=2pt},
    arr/.style={-{Stealth[length=4pt]}, prgray, thick}
]
    \node[block=prblue!10] (hdr) at (0,0) {File Header (magic number, version, stage count)};
    \node[block=prteal!10, below=2pt of hdr] (stg) {Stage Headers (command name, PID for each stage)};
    \node[block=prorange!10, below=2pt of stg, minimum height=2cm] (evt) {Events (timestamp + type + data, repeated)};
    \node[block=prgray!10, below=2pt of evt] (idx) {Index Table (timestamp + file position for each event)};

    \draw[decorate, decoration={brace, amplitude=6pt, mirror}, prgray]
        ($(hdr.north west)+(-0.3,0.1)$) -- ($(idx.south west)+(-0.3,-0.1)$)
        node[midway, left=10pt, font=\small, text=prgray, align=right] {One\\[2pt]\texttt{.prt}\\[2pt]file};
\end{tikzpicture}
\caption{Structure of a \texttt{.prt} trace file.}
\label{fig:traceformat}
\end{figure}

The six event types recorded are:

\begin{itemize}[leftmargin=18pt]
    \item \textbf{PROC\_START} --- a process began running.
    \item \textbf{PROC\_EXIT} --- a process finished (includes exit code).
    \item \textbf{DATA\_OUT} --- a stage wrote data to its output pipe.
    \item \textbf{DATA\_IN} --- a stage received data on its input pipe.
    \item \textbf{PIPE\_EOF} --- a pipe was closed (end of data).
    \item \textbf{PROC\_STATE} --- a snapshot of whether the process is running, sleeping, or blocked.
\end{itemize}

\section{Systems Programming Concepts Used}
\label{sec:concepts}
% ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PipeRewind is a showcase of key systems programming ideas. Here is how
each concept appears in the code:

\renewcommand{\arraystretch}{1.4}
\begin{table}[H]
\centering
\begin{tabularx}{\textwidth}{l X}
\toprule
\textbf{Concept} & \textbf{Where and how it is used} \\
\midrule
\texttt{fork / exec} &
    The capture engine creates a child process for each pipeline
    command with \texttt{fork()}, then replaces it with the real
    program using \texttt{execvp()}. \\
Pipes and \texttt{dup2} &
    Anonymous pipes connect every pair of stages. \texttt{dup2()}
    redirects each child's \texttt{stdin}/\texttt{stdout} to the
    correct pipe end. \\
\texttt{epoll} event loop &
    Instead of checking each pipe one by one (\textit{polling}), the
    engine asks the kernel to wake it up whenever \emph{any} pipe has
    data ready, making it very efficient. \\
\texttt{/proc} filesystem &
    The \texttt{/proc/[pid]/status} file is read periodically to
    check if a process is running, sleeping, or blocked. \\
Threading (\texttt{pthreads}) &
    In live mode, the capture engine runs on a background thread while
    the TUI runs on the main thread, sharing the trace file safely. \\
Binary file I/O &
    The trace writer uses \texttt{fwrite()} for raw binary output, and
    the reader uses \texttt{fseek()} and \texttt{fread()} for random
    access jumps --- much faster than a text format for large files. \\
\bottomrule
\end{tabularx}
\caption{Systems programming concepts in PipeRewind.}
\end{table}

\begin{tipbox}[Record, inspect, and replay]
\begin{lstlisting}[language=bash]
./piperewind record "cmd1 | cmd2 | cmd3"   # save trace

./piperewind live "cmd1 | cmd2"            # record + view

\section{Conclusion}
\label{sec:conclusion}
% ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PipeRewind turns an invisible, one-shot pipeline execution into a
recorded, rewindable experience. Instead of guessing where data was
lost or mangled, you can step through every event, compare outputs
side by side, and watch process states in real time.

The project demonstrates core systems programming skills ---
process creation, pipes, event-driven I/O, threading, and binary
file formats --- all combined into a practical, usable debugging tool.
